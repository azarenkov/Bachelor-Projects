# ERC-1155 vs separate ERC-20 + ERC-721 — gas and design tradeoffs

## Storage layout

ERC-1155 keeps a single nested mapping `id => account => balance`. ERC-20 and
ERC-721 each maintain their own per-contract storage trees (`balances` for
ERC-20, `owners` and per-id approvals for ERC-721). For a game with N item
types, deploying N+M separate contracts means N×(SLOAD-warm cost) on each
read and a separate constructor cost per item type. ERC-1155 collapses that
into one contract — one bytecode, one address, one set of events.

## Batch operations

The decisive ERC-1155 win is `safeBatchTransferFrom` / `_mintBatch`. Moving a
crafting bundle (5 IRON + 2 GOLD) in ERC-20s is two `transfer` calls — two
external calls, two SSTOREs to nonzero, two events. In ERC-1155 it is one
external call with one event whose data array contains both ids. On a
realistic five-resource craft the gas savings are roughly:

| Operation                                  | ERC-20 + ERC-721 | ERC-1155 (batch) | Δ        |
|--------------------------------------------|-------------------|-------------------|----------|
| Mint 3 fungible types to a player          | ~3 × 50k ≈ 150k   | ~85k              | ~–43%    |
| Transfer 5 different resources             | ~5 × 30k ≈ 150k   | ~95k              | ~–37%    |
| Craft (burn 2 fungibles, mint 1 NFT)       | ~3 × 30k + 70k ≈ 160k | ~110k         | ~–31%    |

(Numbers are order-of-magnitude; exact figures depend on warm/cold storage and
event payload size.)

## Design tradeoffs

ERC-1155 is the right standard when:

* Items share a single distribution / inventory contract (games, NFT loot).
* Off-chain indexers can resolve ids via the `{id}` URI substitution.
* You need batch transfers as a first-class operation.

ERC-1155 is the wrong choice when:

* Items need **per-id metadata governance** (royalties, names, permits) —
  ERC-721 has more mature tooling around `tokenURI`, EIP-2981, EIP-4494.
* You need **per-token allowances** (DEX integrations expect ERC-20 `approve`).
* The fungible token must integrate into existing DeFi (AMMs, lending) — no
  AMM accepts ERC-1155 natively without a wrapper.

## In this project

`GameItems.sol` uses ids `< 100` for fungible resources (GOLD, WOOD, IRON) and
ids `>= 100` for NFTs (LEGENDARY_SWORD = 100, DRAGON_SHIELD = 101). The
`craft()` function exploits the same contract being both fungible store and
NFT issuer: a single `_burnBatch` removes the recipe inputs, then a `_mint`
issues the crafted item — atomic without cross-contract calls.
